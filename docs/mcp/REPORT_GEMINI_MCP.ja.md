# MCP トランスポート戦略に関する技術レポート
**作成者:** Gemini CLI
**対象プロジェクト:** FloatingMacro
**日付:** 2026年4月27日

---

## 1. はじめに
現在、MCP クライアント（Claude, Cursor, Gemini 等）とサーバー（FloatingMacro 等）を接続する方式として、**HTTP（SSE）** と **Stdio（標準入出力）** の 2 つが主流となっています。一部で「HTTP 版があれば Stdio 版は廃止すべき」という意見がありますが、macOS ユーティリティのようなローカルアプリケーションにおいては、この両者は**競合ではなく補完関係**にあります。

本レポートでは、なぜ両方のトランスポートを維持することが最適解であるのか、技術的・運用的な視点から論じます。

## 2. 「トランスポートの競合」という誤解の解消
トランスポート（運び方）とロジック（機能）を混同してはいけません。
「HTTP 版があるから Stdio 版はいらない」という主張は、「ブラウザで読めるから API は不要だ」と言うのに似ています。

FloatingMacro の設計において、`fmcli mcp`（Stdio）はアクションのロジックを持たず、リクエストを HTTP API へ転送する **「プロキシ（代理人）」** として実装されています。
*   **ソースの単一性**: 機能の実装（脳）は HTTP API 側に一つしかありません。
*   **不整合の回避**: Stdio 経由でも HTTP 経由でも、最終的に叩かれるコードは同じです。
したがって、メンテナンスコストが倍増するという懸念は、このアーキテクチャにおいては無効です。

## 3. HTTP 時代にあえて Stdio を残す 3 つの合理的理由

### ① クライアントの互換性（Interoperability）
MCP エコシステムはまだ若く、クライアントによって「得意なトランスポート」が分かれています。
*   **Claude Desktop**: ローカルの `config.json` に記述して起動する際、Stdio 方式が最も安定しており、公式の推奨設定です。
*   **Claude Code / Cursor**: HTTP 版をサポートしていますが、プロキシやファイアウォールの設定によっては接続に失敗することがあります。
Stdio 版を提供することは、**「ユーザーがどの AI クライアントを使っていても確実に繋がる」** というアクセシビリティの保証です。

### ② ゼロコンフィグの安定性
HTTP 通信には常に「ポートの衝突」と「認証」の壁があります。
*   もし他のアプリが 17430 番ポートを使っていたら？
*   Bearer トークンの受け渡しに失敗したら？
Stdio 版（`fmcli mcp`）は OS のプロセス管理下で動くため、ポート衝突の影響を受けず、親プロセス（AI クライアント）から直接トークンを渡せるため、接続の確実性が格段に高まります。

### ③ セキュリティ境界の明確化
HTTP サーバーを立てることは、たとえ localhost 限定であっても、ポートをスキャンされるリスクを僅かに伴います。Stdio 方式は「実行権限を持つユーザーが起動したプロセス間」でしか通信が発生しないため、セキュリティモデルがよりシンプルで堅牢です。

## 4. AI エージェント（Gemini CLI）が HTTP MCP を利用する手順

AI エージェントが FloatingMacro の機能を直接 HTTP 経由で活用するためのステップです。

### ステップ 1：ディスカバリー（自己紹介の確認）
まず、認証不要の公開エンドポイントを叩き、アプリの能力を把握します。
```bash
curl -s http://127.0.0.1:17430/manifest | jq
```

### ステップ 2：認証（鍵の取得）
操作系 API に必要な Bearer トークンを macOS Keychain から取得します。
```bash
security find-generic-password -s FloatingMacro -a ControlAPIToken -w
```

### ステップ 3：設定の登録
Gemini CLI 等の `mcpServers` 設定に以下のように記述します。
```json
{
  "mcpServers": {
    "floatingmacro": {
      "httpUrl": "http://127.0.0.1:17430/mcp",
      "headers": {
        "Authorization": "Bearer <TOKEN>"
      }
    }
  }
}
```

## 5. 結論
**HTTP 版があるからといって、Stdio 版を廃止すべきではありません。**

トランスポートは、AI という「新しいユーザー」がアプリにアクセスするための「ドア」です。実装が HTTP API への薄いラップ（プロキシ）で済んでいる現状では、Stdio 版を維持するデメリットはほぼゼロであり、メリット（互換性と確実性）が圧倒的に勝ります。

FloatingMacro は **「世界で最も AI クライアントを選ばないマクロランチャー」** を目指すべきであり、そのためには Stdio/HTTP 両系統のサポートが不可欠であると結論付けます。
